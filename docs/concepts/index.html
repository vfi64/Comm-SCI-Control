Uncertainty Codes (U1-U8)

What: A mandatory label set that marks why certainty is limited.

Why: Prevents hidden guessing and makes risk explicit.

Mini example: "U7: Source A conflicts with Source B."

Common mistake: reporting uncertainty without naming the code.

Execution Pipeline

What: Fixed processing order from parse to render.

Why: Avoids random order changes and improves reproducibility.

Mini example: parse command first, then run preflight before generating output.

Common mistake: mixing rendering decisions before preflight passes.

Preflight Checks (PF-001..008)

What: Validation gates before the first output token.

Why: catches structural failures early (e.g., missing uncertainty line).

Mini example: PF-008 blocks WEB claims without QualityClass.

Common mistake: treating preflight as optional instead of mandatory contract.

SCI Trace

What: Structured reasoning trace (variant-dependent).

Why: improves answer traceability and reviewability.

Mini example: "Plan -> Solution -> Critic -> Learn" in Deep Dive mode.

Common mistake: saying "SCI on" but not selecting a variant letter.

QC Matrix

What: six quality dimensions with score and delta.

Why: gives a compact, comparable quality signal.

Mini example: "Evidence 3 (Δ0), Brevity 2 (Δ-1)".

Common mistake: reporting scores without explaining negative deltas.

Anchor

What: session snapshot for context transfer and re-start.

Why: keeps governance state stable across chat restarts.

Mini example: use Comm Anchor, paste snapshot into new chat.

Common mistake: restarting without re-instantiating JSON + profile state.

Audit

What: compliance check for recent assistant outputs.

Why: detects drift and contract violations post-output.

Mini example: Comm Audit after a long analytical turn.

Common mistake: using audit only when failures are already obvious.

Context-Pressure-Guard

What: threshold-based defense against long-context degradation.

Why: lowers "lost-in-the-middle" risk in long sessions.

Mini example: switch to compact/macro mode at high pressure.

Common mistake: ignoring pressure warnings and continuing in verbose mode.

P0-P5 phases

P0 Parse: commands and control tokens.

P1 Route: map request to workflow path.

P2 State: apply profile/overlays/context state.

P2B Preflight: enforce output prerequisites.

P3-P5: output contract, one-shot repair, final render.

RAG Governance

What: hardening rules for source use (WEB/DOC/TRAIN).

Why: prevents low-quality source mixing and false "green" confidence.

Mini example: R-RAG-001 + PF-008 require WEB QualityClass.

Common mistake: mixing source types without per-claim provenance.

Self-Debunking

What: explicit counter-check of own assumptions and weak points.

Why: reduces overconfidence and makes limitations transparent.

Mini example: add "Self-Debunking" with strongest counterargument.

Common mistake: writing only cosmetic caveats instead of real falsification pressure.
